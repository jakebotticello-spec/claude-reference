# scrub-vN overlay contract — RATIFIED S19 (Jake, 2026-05-30)
**An overlay is a re-scrub: a later, stricter scrubber pass producing a new versioned `records.ndjson` for a snapshot that was already scrubbed at a lower version.** scrub-vN exists for exactly one reason — the scrubber improved, and we want the floor to reflect the better redaction without re-ingesting. This contract was authored in prose and ratified BEFORE any overlay code, because the relationship between an overlay and the sealed snapshot it overlays is immortal-floor-adjacent: a subtle wrong move here writes bad rows that append-only makes permanent. Get the words right first; code obeys the words.
**Invariant 1 — TIGHTEN-ONLY. Raw is never in the picture again.** Raw originals are wiped (Path A, both snapshots, sealed). An overlay therefore CANNOT re-scrub from source — the source is gone. scrub-vN re-scrubs the *already-scrubbed* output of the prior version (scrub-v(N-1)'s records.ndjson). Consequence, stated plainly and accepted: an overlay can redact MORE than its predecessor (catch creds that slipped through), NEVER less, and can NEVER recover text a prior version already redacted or destroyed. If scrub-v1 over-redacted, scrub-v2 cannot give it back — v1's output is all v2 has to work from. Each version is a monotonic tightening of the net over what survived into the prior version's output.
**Invariant 2 — FULL RESTATED STANDALONE. Every scrub-vN is a complete, directly-readable, directly-hashable records.ndjson.** An overlay re-emits EVERY record of the snapshot with the new redactions applied — not a delta of only-changed rows. `scrub-vN/` is a self-contained floor: readable and sha256-verifiable on its own, without reading any other version. This preserves the D9 reframe — the ndjson IS the canonical artifact, a thing you read and hash directly, never a thing you reconstruct at read time. The cost (re-storing unchanged records) is accepted: snapshots are ~370MB, scrub-vN events are rare (only minted on a material scrubber improvement), so this is a handful of full copies over the project's life — disk is cheap, reconstructed-state is not.
**Invariant 3 — ACCRETE FOREVER. A superseded overlay is NEVER removed.** When scrub-v2 is sealed, scrub-v1 stays on disk, sealed, read-only, original hash intact. Overlays appear ALONGSIDE their predecessors; the floor only grows. Two reasons, both binding: (a) doctrine — removing a sealed overlay is a DELETE on the append-only floor, which the S16 `floor_immutable_guard()` triggers forbid with no escape hatch; the architecture already votes against it. (b) audit — scrub-v(N-1) is the only surviving record of what the scrubber did before; it is floor-grade evidence of the scrubber's own lineage, equal to the improvement that superseded it, and does not get summarized away. The max-N glob (v1.5, `_build_seen_set`) reads the highest version as live; the lower versions persist as the verifiable history of how the redaction got there.
**What an overlay MAY touch / MUST NOT touch:** MAY write a new `scrub-vN/` directory under an existing snapshot, containing a full restated records.ndjson + its companion immutable artifacts (scrub-audit, verify.log) for the new pass. MUST NOT modify, delete, or re-hash any existing `scrub-v<N` directory or any file in it — the sealed records.ndjson of every prior version is immutable. MUST NOT touch any raw artifact (there is none — wiped) or rewrite any snapshot's ledger entry retroactively; an overlay is a new layer, not an edit. **The sealed scrub-v1 records.ndjson sha256 MUST read back UNCHANGED after any overlay is generated — this is a NEED-class verification gate (b-ladder rung 6).**
**Append-only applies to overlays themselves:** an overlay is subject to ratify-small-before-scale — generate the first scrub-vN tiny, verify it, Jake ratifies, THEN scale to the full snapshot. Once a scrub-vN records.ndjson is sealed, append-only makes its rows immortal; an un-ratified overlay row is exactly as permanent as a ratified one.
