## INVARIANTS — WITH WHY

# header↔message foreign key — RESOLVED S22 (option (c): drop the cross-table FK, enforce no-orphans at the gate. Ratified by Jake 2026-05-31.)
**THE FINDING.** The S16 schema gave `floor_conv_messages` a composite foreign key `(snapshot_id, conv_uuid)` → `floor_conv_headers(snapshot_id, conv_uuid)`. This was proven 4/4 at S16 — but ONLY against a single-snapshot (baseline) ingest. The first real MULTI-snapshot LOAD (S22) hit a `ForeignKeyViolation` immediately: a delta message for conv `5d0eba7c…` carries `snapshot_id=delta-2026-05-28-…`, but that conv's header was written at BASELINE (`snapshot_id=baseline-2026-05-25-…`) and — per the **no-duplicate-header invariant** — was correctly NOT rewritten in the delta. The composite FK demands message and header share a `snapshot_id`; the no-duplicate-header rule GUARANTEES they will not, for every appended conversation. **The FK encoded an assumption the floor's own ratified invariant makes false.** It was never wrong-on-purpose; it was an untested S16 default the delta exposed the moment both snapshots loaded together.
**THE DECISION — (c): drop the cross-table FK; enforce referential integrity at the load + gate, not in the schema.** Three options were on the table. (a) Re-key the FK to `conv_uuid` alone (would require restructuring the header PK so conv_uuid is globally unique — true under the no-dup-header rule, but it double-encodes that invariant into the schema structure and welds an assumption into immutable bedrock). (b) Write a synthetic duplicate header for appended convs into the delta — **REJECTED, hard: it directly violates the ratified no-duplicate-header invariant** (proven on the floor at S17: the delta is 1,368 lines not 1,371). (c) Drop the cross-table FK entirely; the loader proves "every message's conv_uuid has a header somewhere on the floor" before it commits, and the verification gate re-proves it.
**THE PRINCIPLE (why (c), not just which (c)).** This is the SAME stance the floor already ratified for the message tree at D9: `parent_message_uuid` deliberately has **NO self-referential FK**, with the reasoning written into the lock — *tree integrity is "a PROVEN property of the gate, not a schema-enforced one that could mask floor inconsistency."* A schema constraint enforces a relationship but can also MASK incoherence — it fails one insert without telling you the whole floor is consistent. The floor's philosophy is: prove integrity at the gate, where the whole picture is visible, not at the row level. (c) makes the header↔message link obey the same stance the parent↔child link already does. (a) would split the project's own philosophy — one structural relationship schema-enforced, the other gate-proven, for no principled reason except an S16 accident. **(c) is also the humbler schema**: it claims less, bakes no invariant into the PK structure, and leaves room if we ever learn something new about how snapshots relate — a virtue on a thing that must be immutable. The cost (the DB no longer catches an orphaned message) does NOT vanish; it MOVES to the gate, where the floor keeps all its other integrity proofs.
**THE FIX (S23, loader v1.0→v1.1):** in `seed_shape_load.py`, remove the `CONSTRAINT fk_header FOREIGN KEY …` clause from the `floor_conv_messages` DDL; add a pre-commit orphan-check inside the single transaction — build the set of header `conv_uuid`s, assert every message `conv_uuid` is in it, refuse+rollback with a clear error if any message has no header on the floor. Everything else in the loader is unchanged and proved out at S22 (overlay resolution, counts, single-transaction structure, post-load hardening, owner append-only re-prove). The FK is the only line that moves.
**DO-NOT-RELITIGATE:** (b) is dead (violates a ratified invariant). The choice between (a) and (c) was made on principle (coherence with the D9 parent_uuid precedent + humbler-schema), not on which made the error disappear — all three would. (c) is ratified. Re-opening it should require a NEW reason, not a fresh-eyes re-derivation of the same three options.

# THE PROGENITOR — the law of the pointer catalog (v1 AUTHORED S24 · v2 FOLDED S25, 2026-06-01)

**WHAT A POINTER IS.** A card in a card catalog: a keyword pointing at a span of the immutable floor, laid by a Claude who read the span and judged it worth pointing at. Dict-based, many-to-many (many entries per keyword, many keywords per entry). A **mutable sidecar** that references the floor by key — same architectural slot as pgvector, NEVER a column on the immutable tables (D9).
**TWO KINDS, ONE SPINE.** (1) **FENCE** — points at a decision/constraint where a Claude hitting it cold would *change course*. A fence is NOT a verdict; it is a **LINEAGE** — an ordered chain of (span, date, call, why-as-live-predicate) links, latest current, earlier superseded-not-erased. Claude defends the latest link's *reasoning*, never its *answer* — and reads the chain to tell **evolution** (good — a new why, help add the next link) from **re-litigation** (bad — the same why that already lost, show the fence). The why is a *live predicate* where checkable (1Pass: "still beta? still no invite? RE-CHECK" — a fence that knows why it's a fence can dissolve itself when the world changes). (2) **TEXTURE** — points at a pattern whose *volume is the signal* (representative span + count + spread); calibrates how-to-be-with-Jake, not what-to-decide. Test: does the volume itself tell Claude how to show up? ("fuck" = incidental, no card; Griffin's-pickup = the count IS the meaning, card). Overlap allowed (Griffin is both). **The spine under both — and the floor, and the Claude-lineage — is: append, keep lineage, supersede don't delete.** One architecture, three scales.
**THE BAR.** Default is DON'T point. Earns a card: a fence (changes the next move) or a texture (changes the register). Does NOT: passing mentions, routine build steps, anything already in canon (a fresh Claude reads canon at boot anyway). "Many" = complete over the fences, sparse over the motion — hundreds-to-low-thousands across ~24,138 messages, NOT tens of thousands (tens-of-thousands = the bar slipped, tagging motion). This discipline is *licensed by* corpus-search: completeness lives in the floor, so the catalog is free to be selective. A missing pointer ≠ a missing memory.
**TWO RETRIEVAL MODES.** Pointer-hit = fast, curated, in-session (the card catalog). Corpus-search = slow, exhaustive, separate-window, send-it-and-nap (the stacks themselves) — NOT a mid-session move, like `/jedi-council`. Corpus-search and seeding are the SAME operation at different scales; the slow path's *output is new pointers*. The catalog grows by use.
**DUMB INDEX / SMART READER (the line the whole thing lives on).** The index points; it does not think. It does NOT pre-digest (no summarize/abstract/merge — that corrupts felt-rightness). A selector MAY rank, threshold, cap; MAY NOT rewrite. The catalog points at *settled* judgments (RElitigation-prevention — dumb-faithful) and never predicts *future* relevance (PRElitigation — the index overstepping). Intelligence lives in the reading Claude, never the retriever.
**WHO LAYS THEM.** The reference-file workflow: a Claude proposes, **Jake verifies, commits, pushes** (Claude never claims to have saved). The human is the immutability mechanism. Append-biased, human-gated, fully auditable (history file + changelog; superseded pointers leave a tombstone, never a silent vanish — one Claude's judgment is NEVER enough to kill a reference forever). The original seeding pass is council-laid (too large for per-card review) with Jake spot-auditing the *output shape*; incremental session-pointers get per-card eyes.
**SEEDING GUARDRAILS.** Source is the FLOOR, never chat_search (the prior attempt's failure — chat_search is live + ungrounded; the floor is the immutable ground truth). Not a one-day build. Tree-aware spans only (fanout is a tree-walk up the parent chain / down the branch, NEVER a flat-N timestamp slice that rakes in sibling branches — a contaminated span corrupts felt-rightness). Lay fences as chains even at length-1.
**NAMING:** called "The Progenitor," deliberately NOT "Seed [anything]" — "seed" is reserved in this project for the FLOOR's record-shape (`seed_shape_load.py`, `Seed_Shape_Test_Spec`). The Progenitor is a retrieval-layer thing riding OVER the floor; "seeding" appears only for the act of populating the catalog.

# scrub-vN overlay contract — RATIFIED S19 (Jake, 2026-05-30)
**An overlay is a re-scrub: a later, stricter scrubber pass producing a new versioned `records.ndjson` for an already-scrubbed snapshot.** Authored as law BEFORE any overlay code.
**Invariant 1 — TIGHTEN-ONLY.** Raw is wiped (Path A). An overlay re-scrubs the *already-scrubbed* output of the prior version — redacts MORE never less, never recovers redacted text. Monotonic tightening.
**Invariant 2 — FULL RESTATED STANDALONE.** Every scrub-vN re-emits EVERY record with the new redactions — a complete, directly-readable, directly-hashable records.ndjson, not a delta of changed rows.
**Invariant 3 — ACCRETE FOREVER.** A superseded overlay is NEVER removed (the guard forbids the DELETE + the prior version is the scrubber's own audit trail). The max-N glob reads the highest version as live; lower versions persist as verifiable history.
**MAY/MUST-NOT + the sha gate:** MAY write a new `scrub-vN/` dir under an existing snapshot. MUST NOT modify/delete/re-hash any existing `scrub-v<N` dir. The sealed scrub-v1 sha256 MUST read back UNCHANGED after any overlay (a NEED-class gate). Append-only applies to overlays: ratify-small-before-scale.
